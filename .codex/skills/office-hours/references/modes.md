# Office Hours Modes

## Startup Mode

사업성, 실제 문제, 사용자 행동 근거가 중요한 세션에 사용합니다.

### Operating Stance

- 친절함보다 정확성을 우선합니다.
- 추상어를 그대로 통과시키지 않습니다.
- "관심 있다"와 "진짜 필요하다"를 구분합니다.
- 넓은 플랫폼 비전보다 당장 팔 수 있는 가장 좁은 wedge를 찾습니다.

### Core Questions

아래 질문을 모두 기계적으로 다 물을 필요는 없습니다. 이미 밝혀진 것은 건너뛰고, 가장 큰 불확실성을 먼저 찌릅니다.

1. **Demand reality**: 누가 이게 사라지면 실제로 불편해할까?
2. **Status quo**: 그 사람은 지금 이 문제를 어떻게 억지로 해결하고 있나?
3. **Specific user**: 가장 먼저 붙잡아야 할 실제 사람은 누구인가?
4. **Narrow wedge**: 이번 주 안에 돈을 받거나 강한 사용 신호를 볼 수 있는 가장 작은 형태는 무엇인가?
5. **Observation**: 실제 사용 장면을 봤을 때 예상과 달랐던 것은 무엇인가?

### Output Standard

최소한 아래는 답할 수 있어야 정리 단계로 넘어갑니다.

- 문제는 실제인가, 아니면 아이디어 수준의 상상인가
- 대상 사용자는 카테고리가 아니라 실제 역할로 정의됐는가
- 지금 당장 검증할 가장 좁은 wedge가 있는가
- "이걸 왜 지금 만들지?"에 대한 근거가 있는가

## Builder Mode

사이드 프로젝트, 해커톤, 학습, 오픈소스, 데모 성격의 세션에 사용합니다.

### Operating Stance

- 방향성은 넓게 열어두되 결과물은 빨리 좁힙니다.
- "멋진가?"와 "이번 주 안에 보여줄 수 있는가?"를 함께 봅니다.
- 완벽한 전략보다 실제로 만들고 공유할 수 있는 첫 버전을 우선합니다.

### Core Questions

1. **Coolest version**: 이 아이디어의 가장 재밌고 인상적인 버전은 무엇인가?
2. **Audience**: 누구에게 보여줬을 때 가장 강한 반응이 나올까?
3. **Fastest prototype**: 실제로 써보거나 보여줄 수 있는 가장 빠른 형태는 무엇인가?
4. **Closest existing thing**: 가장 비슷한 기존 도구는 무엇이고, 우리는 어디서 달라지는가?
5. **10x version**: 시간이 무한하다면 무엇까지 키우고 싶은가?

### Output Standard

최소한 아래는 답할 수 있어야 합니다.

- 데모 가능한 첫 버전이 구체적이다
- 보여줄 대상이 선명하다
- 기존 것과 달라지는 지점이 한 문장으로 설명된다
- 이후 확장 방향이 있지만, 첫 버전을 흐릴 정도로 크지 않다

## Fast-Track Rule

사용자가 이미 구체적인 계획과 제약을 제시했다면 질문 단계를 짧게 가져갑니다.

- 남은 불확실성이 큰 질문 1~2개만 확인합니다.
- 그 뒤 바로 premises, approaches, recommendation으로 넘어갑니다.

## Mode Shift Rule

Builder mode 대화 중에 사용자 반응이 사업 검증, 고객, 결제, 팀 운영으로 옮겨가면 Startup mode로 전환합니다. Startup mode 대화 중에 시장성보다 개인적 흥미와 프로토타입 전시가 핵심이면 Builder mode로 완화합니다.
