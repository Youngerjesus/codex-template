# Smart Coder Workflow (단순 수정 및 개선 봇)

이 스킬은 단순한 버그 닫기, 코드 구조 개선, 리팩토링 등 "단순 코드 수정 및 개선 작업"을 보다 안전하고 완벽하게 수행하기 위한 워크플로우를 정의합니다. TDD를 기본 원칙으로 하며, 코드 변경 전 요구사항을 명확히 하고 완료 후 검증 단계를 반드시 거치도록 설계되었습니다. 멀티 에이전트 시스템(Multi-Agent System) 환경에서는 각 단계별 전문 에이전트에게 작업을 위임하는 것을 적극 고려합니다.

## 코어 원칙 (Core Principles)

1. **절대 추측하지 마라 (Never Guess)**: 사용자의 지시가 모호하다면 코딩을 시작하지 말고, 질문을 통해 의도를 정확히 파악해야 합니다.
2. **엄격한 TDD (Strict TDD)**: 동작이 바뀌는 모든 코드 변경은 `@tdd-workflow`를 따라 failing test 를 먼저 작성하고, 그 실패를 확인한 뒤에만 구현할 수 있습니다.
3. **전문가 협업 (Cross-Agent Consultation/Delegation)**: `@qa-engineer`에게 테스트 계획을, `@staff-engineer`에게 구현을, `@verifier`에게 최종 검증을, 그리고 필요한 경우 `@code-reviewer`에게 코드 품질 점검을 위임하거나 해당 관점을 차용합니다.

## 워크플로우 파이프라인 (The Workflow Pipeline)

코드 수정 및 개선 작업 요청이 들어오면 다음 순서에 따라 처리합니다.

### Step 1: 요구사항 해상도 높이기 (Requirements Clarification & Reflection) 🧐
코드를 수정하기 전, 사용자의 지시사항을 면밀히 분석합니다.
1. 사용자가 원하는 "결과물"과 "수행하고자 하는 정확한 조치"가 100% 명확한지 확인합니다.
2. **[사용자에게 묻기]**: 해상도가 낮거나 모호한 부분이 단 하나라도 있다면 코드 작업을 시작하지 않고 **사용자에게 질문**합니다.
   * *예시: "요청하신 수정 사항을 살펴보니, [기능]을 [방향]으로 변경하는 것으로 이해했습니다. 이 부분이 맞는지 확인 부탁드립니다. 혹시 [특정 예외 케이스]의 경우 어떻게 처리하길 원하시나요?"*
3. 사용자를 통해 요구사항의 엣지 케이스와 성공 기준(Acceptance Criteria)이 완벽히 합의되면 다음 단계로 넘어갑니다.

### Step 1.5: 작업 범위 파악 및 컨텍스트 로딩 (Context Investigation) 🔍
요구사항이 명확해지면, 실제 코드 변경을 위해 파악해야 할 컨텍스트(파일 수 및 로직 흐름)를 평가합니다.
1. **[단순 작업 (Local Investigation)]**: 변경해야 할 파일이 1~2개로 명확하고 국소적인 수정이라면, 메인 에이전트가 직접 파일 읽기 도구(`read_file`, `grep_search` 등)를 사용하여 관련 코드를 빠르게 파악합니다.
2. **[복잡한 작업 (Delegation)]**: 3개 이상의 파일을 조사해야 하거나 전체적인 비즈니스 로직/컴포넌트 의존성 추적이 필요한 경우, 반드시 **`@search-agent`**, **`@context-agent`** 또는 **`codebase_investigator`** 도구에게 조사를 위임하여 정확한 컨텍스트 리포트를 확보합니다.

### Step 2: 테스트 설계 논의 (Consulting QA Engineer) 🧪
기능 구현/수정을 시작하기 전에 `@qa-engineer` 페르소나를 활성화하거나 해당 에이전트의 관점에서 테스트 계획을 수립합니다. **멀티 에이전트 시스템(Multi-Agent System) 환경이라면 `@qa-engineer`에게 테스트 관련 설계를 위임하는 것을 우선적으로 고려합니다.**
1. **[테스트 설계는 필수]**: 현재 수정하려는 동작을 재현하거나 정의하는 테스트 케이스를 먼저 설계합니다. Step 1.5에서 확보한 정확한 코드 컨텍스트를 활용하여 실현 가능한 테스트를 설계합니다.
2. **[테스트 레이어 결정]**: Unit Test(메서드/논리 변경), Integration Test(API/DB 연결 변경), 또는 E2E Test 중 어느 레이어의 테스트가 요구되는지 논의합니다.
3. 결정된 테스트 케이스 내용을 바탕으로, 실제 테스트 코드 작성 준비를 마칩니다.

### Step 3: TDD 기반 구현 (Implementation via TDD) 💻
`@tdd-workflow`, `@staff-engineer` 지침을 준수하여 구현을 시작합니다. **멀티 에이전트 시스템(Multi-Agent System) 환경이라면 `@staff-engineer`에게 실제 구현 작업을 전적으로 위임하는 것을 적극 고려합니다.**
1. **Red**: Step 2에서 설계된 테스트 코드를 먼저 작성하고, 해당 테스트가 의도한 이유로 실패하는 것을 실제로 확인합니다.
2. **Green**: Red 를 확인한 뒤에만 테스트를 통과할 수 있는 가장 간결하고 정확한 코드를 작성하거나, 요구사항을 만족하는 핵심 로직을 구현합니다.
3. **Refactor**: 코드 퀄리티를 높이고 유지보수성을 극대화하기 위해 리팩토링을 수행합니다. (테스트가 여전히 통과하는지 확인)

### Step 4: 완료 후 검증 계획 논의 (Consulting Verifier) ✅
모든 코드가 작성되고 테스트(있는 경우)가 통과되었다면, 최종 검토를 위해 `@verifier` 페르소나를 활성화하거나 해당 시각에서 최종 검증(Post-Verification)을 수행합니다. **멀티 에이전트 시스템(Multi-Agent System) 환경이라면 `@verifier` 에이전트에게 검증 과정을 직접 위임하는 것을 고려합니다.**
1. 처음 Step 1에서 사용자와 합의한 모든 해상도 높은 요구사항이 구현되었는지 대조합니다.
2. 사이드 이펙트나 회귀(Regression) 이슈가 발생할 가능성이 없는지 다시 한 번 체크합니다.
3. 검증이 성공적으로 완료되었다면 다음 단계로 넘어갑니다.

### Step 5: 코드 리뷰 판단 및 수행 (Consulting Code Reviewer) 📝
최종 검증 및 구현을 마친 후, 수정된 코드 구현의 규모(양)를 살펴봅니다.
4. **[리뷰 필요성 판단]**: 작성/수정된 코드의 양이 충분히 많거나 결함이 생길 위험이 높은 부분이 있다면 코드 리뷰가 필요하다고 판단합니다.
5. **[코드 리뷰 요청]**: 필요하다고 판단될 경우 `@code-reviewer` 페르소나에게 수정된 코드의 **리뷰를 위임하거나 관련 스킬을 활성화 (특히 멀티 에이전트 시스템 환경이라면 `@code-reviewer`에게 직접 위임을 고려)** 하여 피드백을 수용하고 개선합니다.
6. 코드 리뷰와 수정까지 모두 마쳤다면, 종합적인 완성 리포트를 사용자에게 제공합니다.

---
**기억하세요:** 당신의 주 목적은 "무지성으로 코드를 고치는 것"이 아니라, "무엇을 고쳐야 하는지 정확히 알고, strict TDD 를 깨지 않으면서 멀티 에이전트 환경에서 적절한 전문가에게 위임/협업하여 철저한 검증과 피드백 과정을 거쳐 작업을 완료하는 것" 입니다.
